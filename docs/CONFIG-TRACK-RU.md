# Трекер каталогов и проверка полного восстановления (v5.3)

## Зачем

WAL закрывает базу непрерывно. Но восстановление с нуля на чистом сервере требует ещё и того, что лежит вне базы: `docker-compose.yml`, `.env` с секретами, сертификаты, а у ботов — исполняемый код, миграции, шаблоны, ресурсы. Периодический tar раз в несколько часов давал по этим файлам RPO в часы. Трекер доводит его до минут.

## Модель — та же, что у WAL

Это сделано намеренно, чтобы систему не приходилось держать в голове дважды:

| WAL (база) | Трекер (файлы) |
|---|---|
| `pg_wal` — журнал изменений | git-репозиторий на хосте |
| базовый бэкап | полный bundle в S3 |
| WAL-сегмент | инкрементальный bundle |
| WAL чистится до границы старейшего базового | приращения чистятся до границы старейшего полного |

Приращения дёшевы: в тесте изменение одного `.env` дало bundle в 456 байт против 731 у полного.

## Что отслеживается

Списки берутся из `lib/discovery.sh` — **того же источника, что и бэкапы**, чтобы отслеживаемое и бэкапимое не разъезжались:

- **панель** — `PANEL_ROOT_DIR` целиком (обычно `/opt/remnawave`);
- **боты** — каталог каждого compose-проекта, чей `working_dir` лежит в `/home`, целиком.

Каталоги данных СУБД (`pgdata`, `postgres_data`, `*.rdb` и подобные) исключены по умолчанию: они огромны, меняются постоянно, а главное — уже покрыты WAL и дампами. В тесте это дало репозиторий 224 КБ на проект с 3 МБ данных СУБД. Дополнительные исключения — `TRACK_EXCLUDES`.

## Команды

```bash
rw-backup-full config-track                 # снимок всех проектов
rw-backup-full config-track panel --full    # принудительно новый полный bundle
rw-backup-full config-track --list          # что отслеживается, сколько снимков

rw-backup-full config-restore panel --dest /tmp/check
rw-backup-full config-restore panel --dest /tmp/check --at "2026-07-20 10:00"
rw-backup-full config-restore panel --list  # история снимков
```

Восстановление **никогда** не пишет в рабочий каталог проекта — только в `--dest`. Подстановка на прод остаётся ручным решением оператора.

Расписание — `TRACK_INTERVAL_MIN` (по умолчанию 5 минут). Прогон без изменений занимает доли секунды, поэтому частота почти ничего не стоит.

## Проверка полного восстановления в изоляции

`verify-stack.sh` отвечает на вопрос, ради которого всё строилось: **поднимется ли проект с нуля из одних только бэкапов**. Каталог берётся из трекера, база — из бэкапа, затем поднимается весь стек контейнеров.

```bash
rw-backup-full verify-stack panel
rw-backup-full verify-stack <имя-бота> --keep   # оставить для ручного разбора
```

### Сетевая изоляция — почему это критично

Поднимается настоящая панель с настоящими секретами. Без изоляции копия немедленно повела бы себя как живой сервис: подключилась к реальным нодам Xray и могла их перенастроить, разослала сообщения реальным пользователям в Telegram, полезла в платёжные API, тронула сертификаты. Поэтому изоляция реализована как набор проверяемых гарантий, а не как обещание:

1. Сеть создаётся с `--internal` — выхода в интернет, LAN и на хост нет вообще.
2. Published ports вырезаются — порты хоста не занимаются, конфликта с боевыми сервисами нет.
3. `container_name` вырезается — нет коллизий с работающими боевыми контейнерами.
4. Внешние (`external`) сети и тома вырезаются — копия не подключится к боевой сети и боевым данным.
5. Монтирование `docker.sock` вырезается — иначе контейнер вышел бы за пределы изоляции через API демона.
6. bind-монты внутрь боевых каталогов перенаправляются на восстановленную копию; всё, что осталось снаружи, монтируется только на чтение.
7. `restart: no` — никаких бесконечных перезапусков в песочнице.
8. Сервис БД исключается из compose и заменяется восстановленным экземпляром с сетевыми алиасами по имени сервиса и `container_name`, чтобы приложение нашло его по привычному хосту.

Первые семь пунктов **проверяются самим скриптом** после трансформации: если в итоговой конфигурации остались published ports или `docker.sock`, прогон прерывается как внутренняя ошибка. Дополнительно после подъёма выполняется контрольная проверка: если из контейнера резолвится внешний хост — это фиксируется как нарушение изоляции.

Проверено на реалистичном compose Remnawave (панель + caddy + БД, с `external` сетью и томом, docker.sock и bind-монтами): все девять гарантий подтверждены программными assert'ами.

### Что считается успехом

Все контейнеры стека в состоянии `running`, ни один не в `unhealthy`, перезапусков не больше двух, изоляция не нарушена. Иначе — отчёт с состоянием каждого контейнера и хвостом его логов.

## Какие периоды выставлять

С непрерывным WAL и пятиминутным трекером роль логических бэкапов меняется: они больше не гонка за свежестью, а переносимый снимок «всё целиком» на случай, когда нужно быстро развернуть проект без сборки цепочек. Разумно:

- **логические бэкапы** — раз в 6 часов (`FULL_TIMER_INTERVAL_HOURS="6"`) или дважды в сутки по временам;
- **трекер** — 5 минут (RPO по конфигам и коду);
- **полный bundle трекера** — раз в неделю (`TRACK_FULL_INTERVAL_HOURS="168"`), между ними только приращения;
- **базовый бэкап WAL** — раз в сутки, отправка WAL — раз в минуту.

Отдельно стоит дёргать `rw-backup-full backup-all` перед рискованным деплоем — это надёжнее, чем гнаться за моментом коротким интервалом.

## Компоненты сервера

Серверы разные: где-то нужен полный набор, где-то только логические бэкапы. Что делает конкретный сервер — задаётся одной строкой в едином конфиге:

```bash
FULL_COMPONENTS="panel-backup custom-backup wal config-track metrics"
```

```bash
rw-backup-full components         # что включено
rw-backup-full apply-components   # привести таймеры в соответствие
```

Выключенный компонент не получает таймеров и не выполняет работу, даже если его скрипты установлены; юниты при этом не удаляются — возврат компонента не требует переустановки. Установщик спрашивает набор интерактивно и подставляет разумный по роли (`prod` / `--sandbox`).

## Что убрано

Зависимость от оригинального `distillium/remnawave-backup-restore` удалена полностью: параметр `FULL_PANEL_ENGINE`, автоустановка оригинала, чтение его `config.env` и импорт настроек из него. Панель бэкапится и восстанавливается собственным движком, **формат архива не изменился** — архивы остаются взаимно совместимыми со старым скриптом. Все настройки собраны в одном файле `rw-backup-full.env`.

Если на каком-то сервере оригинальный скрипт всё ещё нужен (например, ради выгрузки в Google Drive, которая не переносилась) — он продолжит работать сам по себе, но rw-backup-full его больше не вызывает и не читает.
