# Fleet-verify: проверка бэкапов всего парка (v5.5)

## Что это

Песочница проверяет бэкапы **каждого подключенного сервера в каждом его хранилище отдельно**: матрица «сервер × S3-бэкенд × категория». Провал в одном хранилище виден, даже если во втором всё хорошо — именно так ловятся «тихо отвалившийся» бэкенд или битые выгрузки в один из бакетов.

## Роль и меню

На сервере-песочнице `FULL_COMPONENTS="sandbox metrics web"` (ставит `install.sh --sandbox`). Интерактивное меню показывает **только** доступные по компонентам разделы: проверка восстановимости, хранилища, настройки, состояние. Пункты бэкапа/WAL на песочнице не отображаются.

Перед проверкой парка или полного стека меню:

1. Актуализирует S3/Telegram-реквизиты (`rw-backup-full sync-creds` → `fleet-creds/<server-id>/`).
2. Для полного стека **требует** ID сервера-источника из fleet (без `--source` на песочнице поднимать нечего).
3. Запускает соответствующий скрипт с выбранной глубиной/режимом БД.

## Откуда берутся параметры: только из веб-сервиса

На песочнице **ничего не настраивается руками** — ни реквизиты хранилищ, ни списки серверов, ни Telegram. Поток:

1. `verify-fleet.sh` (и `sync-fleet-creds`) запрашивает у локального веб-сервиса `/api/fleet/manifest?force=1`.
2. Веб-сервис по SSH собирает с каждого сервера его манифест (`rw-backup-full fleet-manifest`): идентификатор источника, **все S3-бэкенды с реквизитами**, инстансы с типами и параметрами проверок, **настройки Telegram** этого сервера.
3. Песочница строит матрицу и проверяет: скачивает свежайший архив каждой пары «источник × бэкенд × категория» *именно из этого бэкенда*, восстанавливает во временном PostgreSQL и прогоняет проверки профиля.
4. Отчёт по каждому серверу уходит **в Telegram этого сервера**; сводка — в `fleet.json → settings.tg_summary`; **общие ошибки** дополнительно рассылаются по TG **всех** серверов парка.
5. Журнал прогона (`fleet-verify_<host>_<ts>.txt`) выкладывается в S3.

Сводки состояния системы (диски, актуальность, ошибки) — в **09:00 и 21:00** (`rw-status-digest.timer`).

## Профили проверок по типам БД

Проверяемые параметры зависят от типа: `verify-profiles.d/<тип>.env`. Тип инстанса задаётся на сервере в `instances.d/<имя>.env: INST_KIND="panel|bot|site|..."`; панельные архивы проверяются профилем `panel`, архивы ботов — профилем типа соответствующего инстанса.

```bash
PROFILE_REQUIRED_TABLES="public.users public.nodes"  # обязаны существовать и быть непустыми
PROFILE_MIN_TABLES="5"            # минимум пользовательских таблиц
PROFILE_MIN_TOTAL_ROWS="10"       # минимум суммарных строк («пустая, но поднявшаяся» = провал)
PROFILE_CHECK_QUERIES="           # произвольные SQL-проверки: имя::SQL::минимум
активные_ноды::SELECT count(*) FROM nodes::1
"
PROFILE_DEEP_AMCHECK="true"       # pg_amcheck на глубине deep
```

Новый тип (сайт, CRM, что угодно) = новый файл профиля + `INST_KIND` на сервере. Поставляются `panel.env`, `bot.env`, `site.env.example`; неизвестные типы проверяются безопасным минимумом.

## Утилизация ресурсов песочницы

Сервер посвящён только проверкам и управлению, поэтому глубину и параллелизм можно поднимать смело — всё в `fleet.json → settings` (правится через `PUT /api/fleet/settings` или редактором):

```json
"settings": {
  "depth": "deep",
  "parallel": 4,
  "history": 3,
  "schedule": "05:30 17:30"
}
```

`history > 1` отвечает на вопрос «а вчерашний бэкап тоже восстановим?» — важный, когда свежий бэкап уже сделан из повреждённой базы.

## Компактный перенос сервера

```bash
rw-backup-full fleet-pack pack
# на новом сервере после install.sh --sandbox:
rw-backup-full fleet-pack unpack rw-fleet-....tgz
```

Реквизиты хранилищ в файл не входят — они всегда приезжают с серверов автоматически.

## Команды и наблюдаемость

```
rw-backup-full verify-fleet [--server ID] [--backend N] [--depth D]
rw-backup-full verify
rw-backup-full verify-stack panel --source ID [--db-mode dump|base|pitr] [--keep]
rw-backup-full sync-creds
rw-backup-full status-digest
```

Метрики: `rw_fleet_verify_ok{source,backend,category}`, `rw_fleet_server_reachable{server}`.  
Веб-панель: статус серверов (компоненты, бэкапы, диск, WAL, S3), сводка песочницы, история проверок (`/api/verify/history`, кнопка на карточке), API `/api/fleet/overview`.  
`verify-stack --source` предпочитает кэш `fleet-creds/<id>/` после `sync-creds` (веб не обязателен на каждый запуск).

PITR-цепочки тоже в матрице (см. CHANGELOG v5.2.0 про staging WAL). Подтверждено на живом сервере для `INST_KIND=panel`; для `bot`/`site` механизм тот же, но на реальном инстансе бота отдельно не прогонялся.
