# Fleet-verify: проверка бэкапов всего парка (v5.2)

## Что это

Песочница проверяет бэкапы **каждого подключенного сервера в каждом его хранилище отдельно**: матрица «сервер × S3-бэкенд × категория». Провал в одном хранилище виден, даже если во втором всё хорошо — именно так ловятся «тихо отвалившийся» бэкенд или битые выгрузки в один из бакетов.

## Откуда берутся параметры: только из веб-сервиса

На песочнице **ничего не настраивается руками** — ни реквизиты хранилищ, ни списки серверов, ни Telegram. Поток:

1. `verify-fleet.sh` запрашивает у локального веб-сервиса `/api/fleet/manifest`.
2. Веб-сервис по SSH собирает с каждого сервера его манифест (`rw-backup-full fleet-manifest`): идентификатор источника, **все S3-бэкенды с реквизитами**, инстансы с типами и параметрами проверок, **настройки Telegram** этого сервера. Манифест кэшируется (10 мин).
3. Песочница строит матрицу и проверяет: скачивает свежайший архив каждой пары «источник × бэкенд × категория» *именно из этого бэкенда*, восстанавливает во временном PostgreSQL и прогоняет проверки профиля.
4. Отчёт по каждому серверу уходит **в Telegram этого сервера** (настройки из его же манифеста); сводка по парку — в Telegram песочницы, если задан в `fleet.json → settings.tg_summary`.

Добавили сервер в веб-интерфейсе → со следующего прогона его бэкапы проверяются. Сменили ключи S3 на сервере → песочница подхватит автоматически. Единственный секрет, живущий на песочнице, — её SSH-ключ и токен веб-интерфейса.

## Профили проверок по типам БД

Проверяемые параметры зависят от типа: `verify-profiles.d/<тип>.env`. Тип инстанса задаётся на сервере в `instances.d/<имя>.env: INST_KIND="panel|bot|site|..."`; панельные архивы проверяются профилем `panel`, архивы ботов — профилем типа соответствующего инстанса.

```bash
PROFILE_REQUIRED_TABLES="public.users public.nodes"  # обязаны существовать и быть непустыми
PROFILE_MIN_TABLES="5"            # минимум пользовательских таблиц
PROFILE_MIN_TOTAL_ROWS="10"       # минимум суммарных строк («пустая, но поднявшаяся» = провал)
PROFILE_CHECK_QUERIES="           # произвольные SQL-проверки: имя::SQL::минимум
активные_ноды::SELECT count(*) FROM nodes::1
"
PROFILE_DEEP_AMCHECK="true"       # pg_amcheck на глубине deep
```

Новый тип (сайт, CRM, что угодно) = новый файл профиля + `INST_KIND` на сервере. Поставляются `panel.env`, `bot.env`, `site.env.example`; неизвестные типы проверяются безопасным минимумом.

## Утилизация ресурсов песочницы

Сервер посвящён только проверкам и управлению, поэтому глубину и параллелизм можно поднимать смело — всё в `fleet.json → settings` (правится через `PUT /api/fleet/settings` или редактором):

```json
"settings": {
  "depth": "deep",        // quick: целостность+restore; standard: +профильные SQL;
                          // deep: +pg_dumpall всей базы, +pg_amcheck, история архивов
  "parallel": 4,          // одновременных восстановлений (СPU/RAM песочницы — ваши)
  "history": 3,           // проверять не только свежайший, а N последних архивов
  "schedule": "05:30 17:30"   // времена запусков (список любой длины)
}
```

`history > 1` отвечает на вопрос «а вчерашний бэкап тоже восстановим?» — важный, когда свежий бэкап уже сделан из повреждённой базы.

## Компактный перенос сервера

Бэкап песочницы не предполагается: всё её состояние — один компактный `fleet.json` (список серверов + настройки проверок) плюс SSH-ключ и токен. Перенос на новый сервер:

```bash
rw-backup-full fleet-pack pack                 # -> rw-fleet-<host>-<дата>.tgz (~килобайты)
# на новом сервере после install.sh --sandbox:
rw-backup-full fleet-pack unpack rw-fleet-....tgz   # с подтверждением
```

Реквизиты хранилищ и параметры серверов в файл не входят принципиально — они всегда приезжают с серверов автоматически, поэтому переносимый файл мал и не устаревает. Файл содержит секреты (ключ, токен) — обращайтесь как с секретом.

## Команды и наблюдаемость

```
rw-backup-full verify-fleet                     полный прогон матрицы
rw-backup-full verify-fleet --server panel-1    один сервер
rw-backup-full verify-fleet --backend cold-s3   одно хранилище
rw-backup-full verify-fleet --depth deep        разовая глубокая проверка
rw-backup-full verify                           авто: fleet-режим при наличии fleet.json
```

Метрики: `rw_fleet_verify_ok{source,backend,category}` (матрица), `rw_fleet_server_reachable{server}`, счётчики прогона. В дашборде — таймлайн матрицы и статусы доступности серверов. Алерт первой важности: `rw_fleet_verify_ok == 0` — конкретный бэкап в конкретном хранилище не восстанавливается.

PITR-цепочки тоже в матрице: для каждого инстанса каждого сервера, в каждом хранилище, где включена категория WAL, песочница скачивает свежайший базовый бэкап (проверка SHA256), проигрывает весь доступный WAL и поднимает временную БД — с той же логикой, что и ручной `pitr-restore.sh` (включая исключение backup-label файлов из стейджинга, см. CHANGELOG v5.2.0), и с той же проверкой профиля типа. Восстановление выполняется целиком на песочнице — прод не трогается вообще, ни диском, ни CPU. Образ PostgreSQL для fleet-PITR берётся по мажорной версии из метаданных бэкапа (`postgres:<major>`), а не через `docker inspect` живого контейнера источника (у песочницы нет к нему доступа) — если на каком-то сервере используется нестандартный образ (не официальный `postgres`), это стоит проверить отдельно.

⚠️ **Статус проверки: подтверждена только для `INST_KIND=panel`** (сквозной ручной тест на живом сервере: восстановленная копия совпала с боевой базой посекундно). Для `bot`/`site`/прочих типов механизм тот же самый (тот же путь S3, тот же формат бэкапа), но **не проверен на реальном инстансе бота** — соединение `INST_CONTAINER`/`INST_COMPOSE_*` там может отличаться нюансами, которых нет в панели. Fleet-verify печатает предупреждение в лог при первой постановке non-panel PITR-задачи. Прежде чем доверять автоматическим результатам для ботов — прогоните вручную `rw-backup-full pitr-restore <бот> --target latest` хотя бы раз и сверьте count/max по ключевым таблицам, как это было сделано для panel.
