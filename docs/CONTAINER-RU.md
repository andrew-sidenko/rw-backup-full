# Контейнерный режим (v5.1): образ, текущие хосты, подготовка к k8s

Один и тот же образ `rw-backup-full` работает во всех ролях: агент прод-хоста, песочница, веб-сервис, разовый CLI. Сейчас он разворачивается на текущей инфраструктуре через docker compose; при переезде тот же образ станет CronJob'ами в k8s — без переписывания логики.

## 1. Сборка и распространение образа

Из корня репозитория:

```bash
docker build -t rw-backup-full:5.1.0 -f container/Dockerfile .
```

В образе: bash-скрипты проекта, awscli, клиенты PostgreSQL 16 и 17 (PGDG), zstd, age, docker CLI + compose-плагин (демона нет — используется хостовый сокет), supercronic (планировщик), python-окружение веб-сервиса.

Раскатка на парк — через ваш реестр:

```bash
docker tag rw-backup-full:5.1.0 registry.example.com/rw-backup-full:5.1.0
docker push registry.example.com/rw-backup-full:5.1.0
# на хостах:
docker pull registry.example.com/rw-backup-full:5.1.0
```

Нет реестра — перенос файлом: `docker save rw-backup-full:5.1.0 | zstd > rw.tzst`, на хосте `zstd -dc rw.tzst | docker load`.

Обновление версии на хосте = `pull` нового тега + `docker compose up -d` в каталоге агента. Это и есть замена «раскатки скриптов»: один версионируемый артефакт, одинаковый на всех серверах.

## 2. Принцип работы агента на хосте

Контейнер-агент получает хостовый `/var/run/docker.sock` и управляет «соседними» контейнерами точно так же, как это делали скрипты с хоста: `docker exec` в `remnawave-db`/боты, `docker compose` для панели, временные контейнеры восстановления. Расписания внутри контейнера исполняет supercronic; crontab **генерируется из тех же конфигов** (`rw-backup-full.env`, `instances.d/*.env`, включая интервалы и списки времён `*_TIMES`) — настройки не дублируются, меняете конфиг → `docker restart rw-backup-agent`.

**Критично про пути.** Временные контейнеры, которые агент создаёт через хостовый сокет, монтируют каталоги силами *хостового* демона. Поэтому рабочие каталоги обязаны быть смонтированы в агент по тем же путям, что на хосте: `/opt/rw-backup-restore`, `/var/lib/rw-wal`, `/opt/remnawave`, `/home` — слева и справа от двоеточия одно и то же. Compose-файл в `deploy/host/` уже так устроен; не переименовывайте пути.

**Безопасность.** docker.sock внутри контейнера эквивалентен root на хосте. Это осознанная цена: агент по определению управляет контейнерами БД. Компенсации: образ без демонов и лишних сервисов, нет открытых портов, ресурсные лимиты в compose, версионируемый неизменяемый артефакт.

**Что остаётся на хосте.** Клей WAL-слоя — вне контейнера агента по своей природе: `archive_command` исполняется внутри контейнера БД, а `docker-compose.override.yml` панели с монтированием спула — файл на хосте. `wal-enable` из агента настраивает всё это так же, как раньше (сокет + общие пути позволяют), но понимать, где что живёт, важно для отладки.

## 3. Перевод текущего хоста на контейнер

Предпосылка: на хосте уже стоит v5 (конфиги в `/opt/rw-backup-restore`). Поведение, форматы архивов и пути в S3 не меняются — `RW_SOURCE_ID` в `.env` агента ставится равным имени хоста, так что префиксы в бакетах остаются прежними.

```bash
cd rw-backup-full
sudo scripts/host/migrate-to-container.sh
```

Скрипт идемпотентен, каждый шаг — с описанием и подтверждением:

1. Кладёт `deploy/host/docker-compose.yml` + `.env` (с `RW_SOURCE_ID=<hostname>`) в `/opt/rw-backup-restore/agent/`.
2. `docker compose up -d` — агент стартует и печатает сгенерированный crontab в лог.
3. Отключает systemd-таймеры (`rw-backup-full.timer`, `rw-wal-ship@*`, `rw-basebackup@*`, `rw-metrics-export.timer`). Юниты **не удаляются**.

Переходный момент безопасен: пока активны и таймеры, и агент, двойной запуск заданий исключён flock-блокировками (общий `/opt/rw-backup-restore`). Откат одной командой: `sudo scripts/host/migrate-to-container.sh --revert`.

Проверка после миграции:

```bash
docker logs -f rw-backup-agent                      # задания по расписанию
docker exec rw-backup-agent rw-backup-full status --json
docker exec -it rw-backup-agent rw-backup-full      # интерактивное меню
```

Разовые операции без демона (например, на хосте без агента):

```bash
docker run --rm -it \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /opt/rw-backup-restore:/opt/rw-backup-restore \
  -v /var/lib/rw-wal:/var/lib/rw-wal \
  -v /opt/remnawave:/opt/remnawave -v /home:/home \
  -e RW_SOURCE_ID="$(hostname -s)" \
  rw-backup-full:5.1.0 panel-backup
```

## 4. Песочница и веб-сервис в контейнерах

`deploy/sandbox/docker-compose.yml` поднимает два сервиса из того же образа: `rw-sandbox` (verify по расписанию из `SANDBOX_VERIFY_*`) и `rw-web` (порт публикуется только на `127.0.0.1:8787`; наружу — VPN или reverse-proxy с TLS). SSH-ключ и `servers.json` живут в `/opt/rw-backup-restore/web-data` — при первом запуске создайте ключ: `ssh-keygen -t ed25519 -N "" -f /opt/rw-backup-restore/web-data/id_ed25519`, публичную часть разложите по хостам.

Веб-сервис работает с хостами по SSH как раньше; в карточке сервера появилось поле «тип» — `ssh` для текущих хостов, `k8s` зарезервирован (транспорт добавится вместе с переездом, а k8s-источники уже сейчас видны через общую S3-схему, песочницу и Grafana).

## 5. Единая схема источников (готовность к k8s)

Все данные в бакетах лежат по схеме `<prefix>/<категория>/<источник>/...`, где источник — `RW_SOURCE_ID`. Сейчас это имена хостов; при переезде k8s-CronJob'ы этого же образа будут писать `k8s-<кластер>-<нс>` в ту же схему. Следствия, работающие уже сегодня:

- песочница проверяет свежайший логический архив **каждого источника** каждой категории — новые источники подхватываются автоматически, без настройки;
- метрики несут `source`/`kind` (`host`/`k8s`), дашборд показывает таблицу источников и результаты проверок по каждому;
- retention в S3 считается по источнику — переехавший сервис не конфликтует со старым.

Правило на переезд: категория и формат архива сохраняются, меняется только источник. Тогда весь контроль (песочница, дашборд, алерты) не требует изменений.

## 6. Что осталось systemd-вариантом

Контейнерный режим — рекомендация, но не обязанность: `install.sh` со systemd-таймерами полностью поддерживается, конфиги и данные у обоих режимов общие, переключение в любую сторону — `migrate-to-container.sh` / `--revert`. Держать оба режима активными одновременно не нужно (flock спасёт от гонок, но расписания будут дублировать работу).
