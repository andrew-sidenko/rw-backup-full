# Changelog

## v5.5.0 (2026-07-24)

### Исправлено
- **Меню проверки восстановимости**: `ask_choice` переписан на массив вариантов (вместо хрупкого `${!N}`); выбор глубины/охвата/режима БД больше не схлопывается к одному и тому же результату. Для «полного стека» на песочнице сервер-источник выбирается из fleet (обязательный `--source`) — без этого подъём БД/контейнеров не из чего стартовать.
- **`component_enabled` в главном CLI**: `wal-lib.sh` не подключался в `rw-backup-full.sh` — пункты «Компоненты» / `apply-components` падали с `command not found`.

### Добавлено
- **Динамическое меню по `FULL_COMPONENTS`**: на песочнице (`metrics sandbox web`) не показываются бэкап/WAL и прочие нерабочие разделы; пункты нумеруются заново. Недоступные действия скрыты, а не «молча не работают».
- **Синхронизация S3/Telegram с серверов парка** (`sync-fleet-creds` / пункт меню): кэш `fleet-creds/<id>/` обновляется при проверках; уведомления идут по TG реквизитам сервера события.
- **Сводки 09:00 и 21:00** (`rw-status-digest.timer`): диски, актуальность бэкапов, результаты проверок, ошибки → Telegram сервера; на песочнице общие итоги — во все TG парка.
- **Журналы `.txt` в S3** рядом с архивом (то же stem-имя); при ротации удаляются вместе с данными.
- **Веб**: компоненты сервера в статусе, метрики fleet-verify, синхронизация кредов; API `/api/fleet/overview`.

### Изменено
- Установщик ставит/включает юниты по роли и компонентам, включая таймер сводок.
- Общие ошибки fleet-verify дополнительно рассылаются по TG всех серверов.


## v5.4.0 (2026-07-24)

### Добавлено
- **Ротация проверок восстановимости** (`verify-plan.sh`): каждый новый логический дамп и каждый новый базовый бэкап проверяются по одному разу, все остальные прогоны — PITR на точки, **равномерно распределённые по диапазону WAL**. За период цепочка проверяется в разных местах, а не только на краю. Состояние в маркерах — переживает перезапуски; `VERIFY_PLAN_DRYRUN=1` показывает план, не сдвигая ротацию. Тест подтвердил последовательность: dump → base → 4 равномерных PITR-слота → цикл.
- **Подъём БД из базового бэкапа и WAL в verify-stack**: раньше проверялся только логический дамп. Теперь доступны режимы `--db-mode dump|base|pitr` (по умолчанию — автоматическая ротация), восстановление идёт в той же изолированной сети с теми же сетевыми алиасами, так что приложение стека не видит разницы. Стейджинг WAL с исключением backup-label файлов (фикс v5.2.0 перенесён и сюда).
- **Функциональная проверка сервисов**: контейнер `running` больше не считается достаточным. Каждый сервис с портами (для панели — 3000/3001) опрашивается изнутри изолированной сети по имени сервиса; любой валидный HTTP-ответ (включая 401/404) = сервер отвечает, иначе проверяется TCP; логи сканируются на `ECONNREFUSED`/`FATAL:`/ошибки аутентификации БД. Конкретные эндпоинты панели намеренно не зашиты — чтобы проверка не ломалась от версии к версии.
- **Восстановление на чистом сервере** (`scripts/host/bare-restore.sh`): рассчитан на полностью пустую машину — ставит зависимости (docker из официального репозитория, jq/git/rsync/awscli/zstd/age), создаёт каталоги, разворачивает rw-backup-full, запрашивает реквизиты S3, восстанавливает каталог из трекера, поднимает БД и запускает стек. Шесть шагов, каждый с подтверждением и описанием последствий; `--yes` для автоматизации. Шаг запуска явно предупреждает, что стек поднимается БОЕВОЙ, а не изолированный.
- **Юнит периодической проверки полного стека** (`rw-verify-stack.timer`, ежедневно 06:30) с `OnFailure=` уведомлением.

### Изменено
- **Меню полностью переработано**: шесть тематических разделов вместо плоского списка из 18 пунктов, у каждого пункта короткое описание, что он делает. Убраны пункты удалённого distillium-функционала («Установить оригинальный rw-backup», «Backup panel через оригинальный rw-backup»). Параметры запрашиваются интерактивно — выбор инстанса из списка вместо ввода имени, выбор глубины проверки, точки восстановления, режима подъёма БД. Добавлены пункты для трекера каталогов, PITR, диагностики хранилищ, компонентов сервера, отключения старого cron.
- **Все скрипты получают права на запуск при клонировании** (mode 755 в дереве); `install.sh` дополнительно чинит права, если они потерялись при переносе через архив без unix-модов.

Провал проверки данных помечает весь прогон как неудачный (сообщение, метрика, Telegram) — раньше такой случай молча репортился как успех.

### Исправлено
- **Критично, verify-stack.sh: молчаливые сбои без Telegram и без сообщения об ошибке.** У скрипта не было общей защитной сетки — `fail()` (с уведомлением и записью метрики) вызывалась только в явно предусмотренных местах; любой необработанный сбой (`set -e` на непроверенной команде где-то в середине) обрывал выполнение молча — ни строки в консоли, ни Telegram, ни метрики. Добавлен `trap ... ERR` (с `set -o errtrace`, чтобы срабатывал и внутри функций): теперь ЛЮБОЙ сбой — явный или нет — гарантированно даёт сообщение с номером строки и упавшей командой, пишет метрику `rw_stack_verify_ok=0` и уходит в Telegram. Тесты: воспроизведено точное молчание до фикса (несуществующая команда без `fail()`) → теперь полное уведомление; отдельно проверено отсутствие ложных срабатываний на успешном пути (условия в `&&`/`||`/`if` не триггерят перехват, как и положено по семантике bash).
- **verify-stack.sh: `ORIG_ROOT` вычислялся локальным обнаружением контейнеров на песочнице** (`lib/discovery.sh`), которое ничего не находит для проекта на удалённом сервере (обычный сценарий с `--source`) — тихо (без ошибки) подставлялся неверный путь `/opt/<project>` вместо реального `/opt/remnawave`, из-за чего перенаправление абсолютных bind-монтов на восстановленную копию не срабатывало. Для панели используется та же конвенция (`PANEL_ROOT_DIR`), что и в `panel-backup.sh`; для ботов, чей путь при `--source` объективно неизвестен без запроса к самому серверу, добавлено явное предупреждение вместо тихой неверной догадки.

## v5.3.1 (2026-07-24)

Четыре бага, обнаруженные и точно локализованные в эксплуатации.

### Исправлено
- **Критично, весь проект**: системный класс бага "`find`/`du` под `set -o pipefail` валит скрипт целиком, если каталог отсутствует, даже когда `2>/dev/null` скрывает сообщение об ошибке" — не только в `metrics-exporter.sh` (исходный отчёт: экспортер падал на каждом запуске, если WAL для инстанса из `instances.d/` ещё ни разу не включался), но и в `wal-ship.sh` (таймер каждую минуту), и в `rw-backup-full.sh` — `status --json` и `wal-status`, которые дёргает веб-сервис для КАЖДОГО сервера при каждом обновлении статуса в UI. Полный аудит проекта на этот паттерн, точечные `|| true` с явными комментариями во всех найденных местах, регресс-тесты на точное воспроизведение исходной ситуации (инстанс описан, каталог WAL не создан).
- **rw-sandbox-verify.service**: добавлена мягкая ordering-зависимость `After=rw-backup-web.service` (не `Requires=` — юнит веб-сервиса может отсутствовать на хостах без компонента `web`) + retry с backoff (5 попыток, до ~75с) на первый запрос манифеста в `verify-fleet.sh` — `After=` гарантирует только порядок запуска юнитов, не готовность HTTP-сервера принимать соединения, поэтому retry обязателен даже при правильном ordering.
- **Алертинг на отказ проверки парка**: универсальный `notify-unit-failure.sh` + шаблонный юнит `rw-notify-failure@.service`, подключённый через `OnFailure=` к `rw-sandbox-verify.service` и `rw-backup-web.service`. Не зависит от `fleet.json:settings.tg_summary` (частая причина немоты — забыли заполнить): читает Telegram из тех же `FULL_TG_BOT_TOKEN/FULL_TG_CHAT_ID`, что и обычные отчёты о бэкапах, с `tg_summary` как резервным источником и `logger` как последним рубежом.
- **rw-backup-web.service, 226/NAMESPACE**: `mkdir` каталога перед записью юнита защищал только момент установки — при удалении каталога после (например, вручную) сервис вернулся бы в тот же крэш-цикл. Добавлены два независимых уровня: `ExecStartPre=+/usr/bin/mkdir -p ...` в самом юните (`+` обязателен — без него `ExecStartPre` выполняется в уже сандбоксированном окружении и не успевает создать каталог до попытки монтирования `ReadWritePaths`) и `systemd-tmpfiles.d` запись, гарантирующая каталог на каждой загрузке независимо от порядка юнитов.
- Заодно: `install.sh` не копировал `scripts/host/*.sh` в `/opt/rw-backup-restore` вовсе — юнит-нотификатор ссылался бы на несуществующий путь. Добавлено в шаг копирования файлов.

## v5.3.0 (2026-07-23)

### Исправлено
- **Критично, verify-fleet.sh**: скрипт вообще не вызывал `wal_load_full_config` — `rw-backup-full.env` не читался ни разу. Любые настройки в нём (`RW_WEB_URL`, `SANDBOX_AGE_IDENTITY`, `SANDBOX_PG_VERSION` и т.д.) молча игнорировались, откатываясь на дефолты. Внешне выглядело как «правка конфига ни на что не влияет» — например, смена адреса веб-сервиса продолжала долбиться в `127.0.0.1:8787`, даже если в файле стояло другое значение. Добавлен вызов `wal_load_full_config` в начале скрипта, до вычисления `WEB_URL` и любых других переменных. Аудит остальных скриптов подтвердил: это был единственный такой пробел.

### Добавлено
- **Трекер каталогов** (`scripts/track/`): непрерывное отслеживание каталогов проектов ЦЕЛИКОМ — панель (`/opt/remnawave`) и каждый compose-проект из `/home` (у ботов это ещё и исполняемый код, миграции, ресурсы). Модель та же, что у WAL: git-репозиторий на хосте, полный bundle в S3 = базовый бэкап, инкрементальные bundle = сегменты, retention до границы старейшего полного. Каталоги данных СУБД исключены (покрыты WAL/дампами). Восстановление на любой момент: `config-restore --at "дата"`. RPO по файлам — минуты вместо часов.
- **Проверка полного восстановления в сетевой изоляции** (`sandbox/verify-stack.sh`): каталог из трекера + база из бэкапа + подъём всего стека контейнеров. Изоляция как набор проверяемых гарантий: сеть `--internal`, вырезание published ports / container_name / external сетей и томов / docker.sock, перенаправление bind-монтов на копию, read-only для всего вне копии, `restart: no`, отдельный экземпляр БД с сетевыми алиасами. Первые семь гарантий проверяются скриптом после трансформации, плюс контрольная проверка резолва внешнего хоста из контейнера. Без этого поднятая копия панели могла бы перенастроить боевые ноды и разослать сообщения реальным пользователям.
- **Компоненты сервера** (`FULL_COMPONENTS`): один список в едином конфиге определяет, что делает сервер; выключенный компонент не получает таймеров и не выполняет работу. Команды `components` / `apply-components`, интерактивный выбор в установщике с разумным набором по роли.
- `lib/discovery.sh` — единый источник списка проектов для бэкапов, трекера и песочницы.

### Изменено
- **Зависимость от distillium/remnawave-backup-restore убрана полностью**: удалены `FULL_PANEL_ENGINE`, автоустановка оригинала, чтение его `config.env` и импорт настроек. Формат архива панели не изменился — взаимная совместимость сохранена.
- **Конфигурация консолидирована в один файл**: настройки, которые раньше подтягивались из `config.env`, переехали в `rw-backup-full.env`; файл переструктурирован по разделам.


## v5.2.0 (2026-07-23)

### Исправлено
- **Fleet-verify: PITR добавлена в матрицу проверок.** Раньше `verify-fleet.sh` проверял только логические архивы (panel/custom-bot); PITR-цепочки (базовый бэкап + WAL) теперь проверяются по той же матрице «сервер × хранилище × инстанс» — для каждого инстанса каждого сервера, в каждом хранилище с включённой категорией WAL. Восстановление целиком на песочнице (прод не нагружается), с той же логикой стейджинга WAL, что и pitr-restore.sh (включая фикс backup-label-коллизии ниже), и с тем же профилем проверок по типу инстанса. Изолированный тест на симулированном S3-бакете с реальной .backup-коллизией подтвердил: реальные сегменты стейджатся побайтово корректно, backup-label не создаёт отдельного файла и не подменяет сегмент.
- **Критично, pitr-restore.sh**: подготовка WAL к восстановлению могла подменять настоящий сегмент его же backup-label файлом (`<segment>.<offset>.backup`, создаётся при каждом базовом бэкапе). Оба файла делят первые 24 символа имени (имя сегмента), `stage_one()` наивно брал их как разделитель, backup-label (несколько сотен байт) сортировался раньше и занимал место под именем сегмента — настоящий 16-МБ сегмент затем молча пропускался проверкой идемпотентности. Итог: PostgreSQL при восстановлении падал с `archive file "..." has wrong size: 347 instead of 16777216`. Сам архив на диске не повреждался — баг был только в стейджинге для восстановления. Исправлено: `.backup`-файлы явно исключены из стейджинга (PostgreSQL их и не запрашивает через restore_command). Регресс-тест: реальный сегмент теперь стейджится побайтово корректно при наличии одноимённого backup-label.
- **pitr-restore.sh**: при падении временного контейнера восстановления диагностика была пустой — контейнер поднимался с `--rm`, при крахе Docker удалял его мгновенно, и последующий `docker logs` бил в "No such container" (ошибка глушилась). `--rm` снят (уборка — явным `docker rm`/trap), при падении печатаются логи, exit code, OOMKilled и подсказка по правам PGDATA. Заодно: владелец PGDATA для восстановления больше не угадывается как uid 999, а выясняется у фактического образа (`docker run --entrypoint id`) — частая причина мгновенного краха на нестандартных postgres-образах.
- **install-web.sh**: юнит rw-backup-web.service падал с 226/NAMESPACE (рестарт-цикл, сервис не отвечал) на серверах без node_exporter — ReadWritePaths=/var/lib/node_exporter требует существования каталога на момент старта systemd, а создавался он только сторонними скриптами метрик. Установщик теперь создаёт /var/lib/node_exporter/textfile_collector сам и проверяет, что сервис реально поднялся, показывая journalctl при провале.
- **Критично, basebackup**: ложный провал валидации «в архиве нет PG_VERSION» на больших БД — grep -q закрывал конвейер после совпадения, tar/декомпрессор получали SIGPIPE, и под set -o pipefail статус конвейера становился ошибочным при полностью исправном бэкапе. Валидация переведена на «листинг → файл → grep»; главный критерий — global/pg_control; при провале архив сохраняется как *.INVALID со списком членов для разбора. Тот же паттерн устранён в panel-backup.
- enable-archiving: явная передача docker-compose.override.yml в compose (явный -f отключает автоподхват override), контроль фактического пересоздания контейнера по ID с --force-recreate, диагностика отсутствующего монтирования и предупреждение о копящемся pg_wal при падающей archive_command.
- install-web: проверка ensurepip до создания venv, пересоздание неполного venv (--clear); install.sh: обновление web-src копированием содержимого (повторный запуск не вкладывал каталог внутрь).

### Добавлено
- **Fleet-verify** (verify-fleet.sh): проверка бэкапов каждого сервера × каждого S3-хранилища × каждой категории отдельной проверкой; параметры (реквизиты хранилищ, инстансы, Telegram) извлекаются из веб-сервиса автоматически (`/api/fleet/manifest` ← `rw-backup-full fleet-manifest` по SSH, кэш). Персональные Telegram-отчёты серверам + сводка песочницы; метрики rw_fleet_verify_ok{source,backend,category} и rw_fleet_server_reachable; 2 новые панели дашборда.
- **Профили проверок по типам БД** (verify-profiles.d/): panel, bot, site.example; обязательные таблицы, минимумы таблиц/строк, произвольные SQL-проверки, deep-опции. Тип задаётся INST_KIND.
- **Утилизация ресурсов песочницы**: settings в fleet.json — depth (quick/standard/deep: +pg_dumpall, +pg_amcheck), parallel (одновременные восстановления), history (N последних архивов на источник), schedule.
- **fleet.json** — единственный компактный файл состояния сервера песочницы (миграция из servers.json автоматическая); **fleet-pack pack/unpack** — перенос сервера одним файлом (fleet.json + SSH-ключ + токен), бэкап сервера не требуется.
- verify-entry.sh: `verify` автоматически использует fleet-режим при наличии fleet.json (systemd и контейнер).

## v4.0.0 (2026-07-22)

Аддитивный слой поверх v3 — все прежние команды, форматы архивов и конфиги без изменений.

### Добавлено
- **WAL-архивация PostgreSQL** для панели и кастомных ботов: archive_command → спул на хосте → сжатие (zstd/gzip) → локальный архив + S3. archive_command не ходит в сеть — сбой S3 не роняет БД.
- **Базовые бэкапы `pg_basebackup`** по расписанию с метаданными (SHA256, стартовый сегмент, timeline) и выгрузкой в S3. Периодичность per-instance (`INST_BASEBACKUP_INTERVAL_HOURS`, `INST_WAL_SHIP_INTERVAL_MIN`, `INST_ARCHIVE_TIMEOUT`).
- **PITR-восстановление** (`pitr-restore`): из локального архива или S3, на последнюю точку / момент времени / LSN / только полный бэкап; безопасный режим во временном контейнере и деструктивный `--in-place` с сохранением старого PGDATA.
- **Песочница** (`verify`, `install.sh --sandbox`): ежедневная автоматическая проверка восстановимости на отдельном сервере — PITR-цепочки (контрольные таблицы, полный pg_dumpall) и логические архивы панели/ботов из S3.
- **Безопасный retention**: WAL удаляется только до границы старейшего хранимого базового бэкапа, отдельно локально и в S3.
- **Метрики** textfile collector для VictoriaMetrics/Grafana + Telegram-уведомления.
- Управление: `wal-enable/wal-disable` (compose override + ALTER SYSTEM + сквозной тест), `wal-status`, `wal-timers`, шаблонные systemd-юниты с per-instance интервалами.
- Опциональное age-шифрование базовых бэкапов и WAL (приватный ключ на проде не хранится).
- **Гибкие расписания** для всех таймеров (базовые бэкапы, WAL-шиппер, логические бэкапы v3, песочница): либо интервал, либо список конкретных времён произвольной длины (`*_TIMES="03:00 12:30 21:00"`) с опциональным часовым поясом; реализация через systemd OnCalendar.

### Совместимость
- Оригинальный distillium config.env читается только на чтение; форматы `remnawave_backup_*.tar.gz` и `custom_bot_*.tar.gz` не изменены.
- Не поддерживается автоматический `wal-enable` для Patroni (защита от перезаписи конфига; см. README-RU, «Устранение неполадок»).
